Quark starts with a simple irritation: software tools often become larger than the problem they are meant to expose.

The project asks whether a smaller system can still feel serious. Not toy-small, but instrument-small: limited enough that the edge of the system remains visible while you work.

## Working Constraint

A useful tool should make its constraints legible. When the constraint is hidden, the user has to reverse-engineer the machine before they can think through it.

Quark should make the cost of every abstraction visible.

```txt
input -> model -> surface -> feedback
```

## Open Question

The hard part is not minimalism by removal. The hard part is deciding which pieces deserve to exist.
